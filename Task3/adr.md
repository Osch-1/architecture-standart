### **Функциональные требования**

|**№**|**Действующие лица или системы**|**Use Case**|**Описание**|
| :-: | :- | :- | :- |
| UC1 | Посетитель сайта, Сайт, Интернет-банк | Просмотр депозитов на сайте | Посетитель открывает страницу депозитов на сайте. Сайт запрашивает актуальные ставки у микросервиса депозитов ИБ и отображает список доступных продуктов (F1) |
| UC2 | Посетитель сайта, Сайт, Система кол-центра | Подача заявки через сайт | Посетитель заполняет форму (ФИО, телефон), заявка передаётся в систему кол-центра. Оператор связывается с клиентом и предлагает условия (F2, F3) |
| UC3 | Клиент ИБ, Интернет-банк | Просмотр персонализированных ставок | Клиент входит в интернет-банк и видит список депозитов с персональными ставками из кэша микросервиса депозитов (F4, P5) |
| UC4 | Клиент ИБ, Интернет-банк, Брокер сообщений, АБС, СМС-шлюз | Подача заявки на депозит в ИБ | Клиент выбирает депозит, указывает счёт и сумму, подтверждает СМС-кодом. Заявка асинхронно отправляется через брокер в АБС (F5, F6, R4, R5) |
| UC5 | Менеджер бэк-офиса, АБС | Обработка заявки на депозит | Менеджер видит заявку в АБС, согласовывает ставку, подтверждает условия и открывает депозит. Статус заявки возвращается в ИБ через брокер (F8, +R8) |
| UC6 | АБС, СМС-шлюз, Телеком-оператор | Уведомление клиента по СМС | После согласования ставки и после открытия депозита АБС отправляет клиенту СМС-уведомления (F9, F10) |
| UC7 | Кредитный аналитик, Менеджер бэк-офиса, АБС | Управление ставками в АБС | Кредитный аналитик ведёт базовые ставки, менеджер бэк-офиса назначает спецставки — всё в АБС вместо Excel (F7) |
| UC8 | АБС, Брокер сообщений, Микросервис депозитов | Синхронизация ставок и статусов | АБС публикует обновлённые ставки и статусы заявок через брокер, микросервис депозитов обновляет кэш и отображает актуальный статус клиенту (P5, R4) |
| UC9 | Клиент, Менеджер фронт-офиса, АБС | Открытие депозита в отделении | Клиент приходит в отделение, менеджер фронт-офиса оформляет заявку на депозит в АБС. Бэк-офис обрабатывает заявку по стандартному процессу (F8, +R8) |

### **Нефункциональные требования**

|**№**|**Требование**|
| :-: | :- |

| R2, R3 | Переключение на резервный ЦОД при сбое. Сервисы ИБ должны оставаться доступными |
| R4, R5 | Асинхронное взаимодействие ИБ и АБС через промежуточный слой (брокер сообщений). Запрещено прямое обращение ИБ к API/БД АБС |
| +R3 | Текущая платформа ИБ (.NET Framework 4.5) несовместима с Kafka — необходим альтернативный брокер |
| P1 | Время отклика по всем операциям — миллисекунды |
| P2 | Устранить проблему медленной загрузки справочных данных (сейчас >1 сек) |
| P3 | Горизонтальное масштабирование и балансировка нагрузки |
| P4 | АБС масштабируется только вертикально (Oracle PL/SQL) |
| P5 | Кэширование ставок и справочных данных на стороне ИБ для ускорения отклика |
| U1, U2 | Удобный интерфейс, соблюдение корпоративного дизайна. Микросервис депозитов должен вписываться в существующий UI монолита |
| S1 | Использование существующих технологий банка (MS SQL, Oracle, .NET) |
| S2 | Новые технологии должны быть совместимы с существующими платформами разработки |
| S3 | Внутренняя экспертиза по используемым технологиям (снижение зависимости от подрядчиков) |
| S4 | Использование Kafka для очередей сообщений на перспективу (стратегическое направление) |
| S5 | Начало перехода ИБ на микросервисную архитектуру с функционала депозитов |
| S6 | Документирование для дальнейшего расширения (кредитование и другие продукты) |
| S7 | Функционал СМС реализуется силами команды банка |
| S8 | Версионирование API новых микросервисов |
| +R7 | Новые клиенты не могут быть приняты без личной проверки документов (регуляторное ограничение) |

### **Решение**

#### Диаграммы

Диаграмма контекста (C4 System Context): [c4_context.puml](c4_context.puml)

Диаграмма контейнеров (C4 Container): [c4_container.puml](c4_container.puml)

#### Ключевые решения

**1. RabbitMQ как брокер сообщений**

Для асинхронного взаимодействия между интернет-банком и АБС выбран RabbitMQ. Kafka — стратегически предпочтительный вариант (S4), но текущая платформа ИБ на .NET Framework 4.5 с ним несовместима (+R3). RabbitMQ имеет зрелые клиентские библиотеки для .NET и полностью решает задачу асинхронной развязки (R4, R5). Для обеспечения SLA 99,9% (R1) RabbitMQ разворачивается в кластерной конфигурации с зеркалированием очередей между основным и резервным ЦОД (R2). Миграция на Kafka запланирована после модернизации платформы ИБ.

**2. Микросервис депозитов (.NET, MS SQL)**

Новый микросервис создаётся рядом с существующим монолитом ИБ (S5). Монолит делегирует депозитные операции микросервису через REST API. Используется .NET — технология, которую знает команда банка (S3), и MS SQL — СУБД из существующего стека (S1). Микросервис отвечает за логику подачи заявок, взаимодействие с брокером, кэширование ставок (P5) и предоставление API ставок для сайта (F1). UI депозитных операций встраивается в существующий интерфейс монолита для соблюдения корпоративного дизайна (U1, U2). API микросервиса версионируется (S8). Микросервис поддерживает горизонтальное масштабирование (P3).

**3. Адаптер сообщений на стороне АБС (Java)**

Легковесный Java-сервис на стороне АБС принимает и отправляет сообщения через RabbitMQ, вызывая PL/SQL-процедуры в Oracle. В команде АБС есть специалисты с опытом в Java (+R4), что позволяет разработать и поддерживать адаптер собственными силами. Через адаптер возвращаются как обновлённые ставки, так и статусы обработки заявок — клиент видит актуальный статус в ИБ.

**4. Кэш ставок на стороне ИБ**

Ставки и справочные данные кэшируются в БД микросервиса депозитов (MS SQL) и обновляются асинхронно через брокер при изменении ставок в АБС. Это устраняет проблему медленной загрузки справочных данных (P2), обеспечивает время отклика в миллисекунды (P1) и снижает нагрузку на АБС. Кэш также используется сайтом для отображения витрины депозитов (F1).

**5. Ведение ставок в АБС вместо Excel**

Управление ставками переносится из Excel-файлов в АБС (F7). Кредитный аналитик и менеджер бэк-офиса работают со ставками в едином интерфейсе АБС. Это устраняет ручной email-обмен файлами и обеспечивает единый источник данных.

**6. Интеграция сайта с системой кол-центра и ИБ**

Заявки с сайта передаются в систему кол-центра по REST API (F2, F3). Для отображения витрины депозитов (F1) сайт запрашивает актуальные ставки у микросервиса депозитов ИБ через REST API.

**7. СМС-функционал**

СМС-код подтверждения при подаче заявки в ИБ (F6) отправляется микросервисом депозитов через СМС-шлюз. СМС-уведомления о согласовании ставки (F9) и открытии депозита (F10) отправляются со стороны АБС через адаптер сообщений. Разработка ведётся командой банка (S7).

**8. Офлайн-сценарий (отделения)**

Процесс открытия депозита в отделении сохраняется: менеджер фронт-офиса оформляет заявку напрямую в АБС. В MVP заявки из отделений обрабатываются бэк-офисом по существующему процессу (+R8).

**9. Аудит и логирование**

Все операции по заявкам на депозиты логируются в микросервисе депозитов (на стороне ИБ) и в адаптере сообщений (на стороне АБС) для обеспечения банковского комплаенса и расследования инцидентов (F13).

### **Альтернативы**

**1. Kafka вместо RabbitMQ.** Стратегически предпочтительный вариант, но несовместим с текущей платформой ИБ на .NET Framework 4.5 (+R3). Потребовалось бы сначала обновить всю платформу, что выходит за рамки MVP.

**2. Синхронный REST API между ИБ и АБС.** Проще в реализации, но нарушает требования R4/R5. Прямые вызовы создадут нагрузку на Oracle DB (P4) и жёсткую связанность систем.

**3. Улучшение текущего DB Link.** Сохранение существующей схемы прямого подключения к БД АБС с добавлением кэширования. Прямо противоречит R4 — DB Link уже выявлен как проблема в текущей архитектуре.

**4. ESB (Enterprise Service Bus).** Избыточное решение для текущего масштаба. Вносит новую сложную технологию без внутренней экспертизы (нарушает S3).

**5. Полный реврайт ИБ на современный стек (.NET 6+).** Позволил бы использовать Kafka и современные паттерны, но требует значительных сроков и бюджета, несовместим с целью MVP за 6 месяцев. Высокие риски при полной замене работающей системы.

**Недостатки, ограничения, риски**

| Категория | Описание |
| :- | :- |
| Двойная работа по миграции | RabbitMQ — тактический выбор. При модернизации ИБ потребуется миграция на Kafka, что означает повторную работу по интеграции |
| Согласованность данных | Кэш ставок на стороне ИБ обновляется асинхронно — возможна кратковременная рассинхронизация. Клиент может увидеть устаревшую ставку. Риск минимизируется частотой обновления и финальной проверкой ставки при подтверждении заявки в АБС |
| Новая точка отказа | RabbitMQ — новый компонент инфраструктуры. Требуется кластеризация для обеспечения отказоустойчивости (R1, R2) |
| Сложность на стороне АБС | Адаптер сообщений — новый компонент, который нужно разрабатывать и поддерживать командой АБС |
| Ручная обработка в MVP | В MVP менеджер бэк-офиса обрабатывает заявки вручную (+R8), что ограничивает пропускную способность процесса |
| Регуляторные ограничения | Новые клиенты не могут открыть депозит онлайн без личной верификации (+R7) — функционал доступен только существующим клиентам ИБ |
| Зависимость от подрядчика ИБ | Изменения в монолите ИБ (интеграция с микросервисом) могут потребовать координации с подрядчиком (+R1) |
| Нагрузка на команды | Команда ИБ (10 внутренних + 10 подрядчика) и команда АБС (20 человек) должны параллельно разрабатывать микросервис депозитов, адаптер сообщений и внедрять RabbitMQ. Для небольших команд это существенная нагрузка, требующая чёткого планирования и приоритизации |
