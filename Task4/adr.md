### **Название задачи:** Обеспечение доступа к ставкам по депозитам для кол-центра и партнёрского кол-центра
### **Автор:** Герасимов Даниил
### **Дата:** 2026-03-22
### **Статус:** Предложено

### **Контекст**

Банк «Стандарт» запускает маркетинговую кампанию для привлечения новых и текущих клиентов к депозитным продуктам. Основная категория клиентов банка — люди старшего возраста, которым может быть непросто сориентироваться в новых онлайн-процессах. Это создаёт риск значительного роста числа звонков в кол-центр с вопросами об условиях депозитов.

Сейчас операторы кол-центра не имеют доступа к актуальным ставкам по депозитам и не могут проконсультировать клиентов. Ставки ведутся в АБС (решение MVP, Task3). Для разгрузки кол-центра необходимо также подключить партнёрский кол-центр (100 операторов), но его внешняя система не поддерживает API-вызовы — доступен только файловый обмен через SFTP.

Данное решение строится на базе архитектуры MVP онлайн-открытия депозитов (Task3) и расширяет сервис интеграции депозитов АБС новым функционалом.

Оператор видит список депозитных продуктов с текущими ставками и консультирует клиента по условиям открытия депозита, срокам и процентным ставкам |
| UC2 | Кредитный аналитик, АБС, Сервис интеграции депозитов АБС, SFTP-сервер | Обновление ставок и автоматическая выгрузка на SFTP | Кредитный аналитик обновляет ставки в АБС. Система партнёрского кол-центра периодически забирает файл со ставками с SFTP-сервера банка по протоколу SFTP. ### **Решение**

#### Диаграммы

Диаграмма контекста (C4 System Context): [c4_context.puml](c4_context.puml)

Диаграмма компонентов (C4 Component): [c4_component.puml](c4_component.puml) — детализация сервиса интеграции депозитов АБС и системы кол-центра

#### Ключевые решения

**1. Сервис интеграции депозитов АБС (расширение адаптера сообщений)**

Существующий адаптер сообщений на стороне АБС (Java), разработанный в рамках MVP для взаимодействия с RabbitMQ, расширяется новым функционалом — REST API для предоставления актуальных ставок и модулем генерации CSV-файлов. Сервис читает ставки напрямую из БД АБС (Oracle) — единого источника данных. Это обеспечивает актуальность ставок без задержек, связанных с асинхронной репликацией через брокер.

Команда АБС (20 человек) уже имеет экспертизу в Java и поддерживает адаптер сообщений, что минимизирует затраты на внедрение. REST API версионируется (NR6) для безопасного развития контракта.

**2. REST API ставок для системы кол-центра**

Система кол-центра (Java Spring Boot) получает ставки через REST API сервиса интеграции депозитов АБС. Для этого в бэкенде кол-центра добавляется модуль-клиент REST API, а в UI (React.js) — новый экран отображения ставок оператору. Ставки могут кэшироваться на стороне системы кол-центра (в PostgreSQL или in-memory) для повышения отказоустойчивости.

Система кол-центра — подрядная платформа (Java Spring Boot, PostgreSQL, React, микросервисная архитектура, +R4), доработка выполняется командой подрядчика (5 человек).

**3. SFTP-сервер для передачи файлов партнёру**

В DMZ банка разворачивается SFTP-сервер (OpenSSH или аналог). Сервис интеграции депозитов АБС генерирует CSV-файл со ставками и выгружает его на SFTP при каждом обновлении ставок (NR2). Партнёрский кол-центр забирает файл по протоколу SFTP.

Безопасность обеспечивается:
- Аутентификация по SSH-ключам (NR3)
- Ограничение доступа по IP-адресам партнёра (NR3)
- Размещение в DMZ — изоляция от внутренней сети банка
- Партнёр имеет доступ только на чтение (read-only)

**4. Передача информации партнёрскому кол-центру через файлы**

Партнёрский кол-центр работает во внешней информационной системе и не может принимать API-вызовы. Единственный доступный канал — файлы через SFTP. CSV-файл содержит: название депозитного продукта, срок размещения, минимальную сумму, процентную ставку, дату актуальности. Партнёр забирает файл по согласованному расписанию.

**5. Сохранение текущего процесса работы со ставками**

Процесс ведения ставок не меняется — кредитный аналитик и менеджер бэк-офиса управляют ставками в АБС (решение F7 из MVP). Сервис интеграции читает ставки из БД АБС и предоставляет их потребителям (система КЦ, SFTP). Это обеспечивает единый источник данных и не требует изменений в бизнес-процессе.

#### Список крупных задач по системам

**АБС / Сервис интеграции депозитов (Java):**
1. Разработать REST API для получения актуальных ставок по депозитам (эндпоинт GET /api/v1/rates)
2. Реализовать модуль генерации CSV-файла со ставками
3. Реализовать выгрузку CSV-файла на SFTP-сервер (SFTP-клиент)
4. Настроить триггер обновления: при изменении ставок в АБС + периодический (1 раз в час)
5. Добавить логирование и аудит операций (NR7)
6. Версионирование API (NR6)

**SFTP-сервер (инфраструктура):**
1. Развернуть SFTP-сервер в DMZ банка
2. Настроить учётные записи: запись для сервиса интеграции АБС, чтение для партнёра
3. Настроить SSH-ключи и ограничение по IP для партнёра
4. Настроить мониторинг и ротацию файлов/логов

**Система кол-центра (Java Spring Boot, подрядчик):**
1. Разработать модуль-клиент REST API сервиса интеграции АБС для получения ставок
2. Реализовать кэширование ставок (опционально, для отказоустойчивости)
3. Разработать UI-экран отображения ставок оператору (React.js)
4. Интеграционное тестирование с REST API сервиса интеграции

**Система партнёрского кол-центра (внешний партнёр):**
1. Настроить SFTP-клиент для подключения к SFTP-серверу банка
2. Интегрировать загруженный CSV-файл в рабочий процесс операторов
3. Согласовать расписание получения файлов

### **Альтернативы**

**1. Микросервис депозитов ИБ как единый источник ставок.** Микросервис депозитов интернет-банка уже имеет кэш ставок и REST API (используется сайтом). Можно было бы переиспользовать его для системы кол-центра и добавить модуль экспорта на SFTP. Преимущества: единый источник данных для всех потребителей, переиспользование существующего API. Недостатки: создаёт зависимость системы кол-центра от интернет-банка; кэш обновляется асинхронно через RabbitMQ — возможна задержка; увеличивает нагрузку на микросервис депозитов, который и так обслуживает ИБ и сайт.

**2. Прямой доступ системы кол-центра к БД АБС.** Система кол-центра могла бы читать ставки напрямую из Oracle через DB Link или JDBC. Преимущества: минимум новых компонентов. Недостатки: нарушает принцип изоляции систем (R4, R5); создаёт нагрузку на Oracle DB (P4); жёсткая связанность.

**3. Выгрузка файлов вручную (email).** Сотрудники банка могли бы отправлять файл со ставками партнёру по email. Преимущества: не требует технической реализации. Недостатки: ручной процесс, подвержен ошибкам; нет гарантии актуальности; не масштабируется.

**4. Отдельный микросервис экспорта ставок.** Новый сервис, подписанный на обновления ставок через RabbitMQ, предоставляющий API и генерирующий файлы. Преимущества: чистое разделение ответственности. Недостатки: новый компонент для разработки и поддержки; дублирует логику кэширования ставок; избыточно для текущего масштаба.

**Недостатки, ограничения, риски**

| Категория | Описание |
| :- | :- |
| Устаревание данных в файле | CSV-файл на SFTP обновляется периодически. Между обновлениями партнёрский кол-центр может работать с неактуальными ставками. Риск минимизируется частотой обновления (при каждом изменении + не реже 1 раза в час) и указанием даты актуальности в файле |
| Нагрузка на Oracle DB | REST API сервиса интеграции читает ставки напрямую из Oracle. При высокой частоте запросов от системы кол-центра возможна дополнительная нагрузка на БД (P4). Рекомендуется кэширование на стороне сервиса интеграции (in-memory) |
| Безопасность SFTP | SFTP-сервер в DMZ — точка входа из внешней сети. Необходимы строгие политики безопасности: SSH-ключи, ограничение по IP, мониторинг подключений, ротация ключей |
| Зависимость от подрядчика кол-центра | Доработка системы кол-центра (REST-клиент, UI ставок) выполняется командой подрядчика (5 человек). Необходима координация сроков и приоритетов |
| Новый компонент в инфраструктуре | SFTP-сервер — новый элемент, требующий администрирования, мониторинга и поддержки со стороны IT-отдела |
| Масштабируемость SFTP | Файловый обмен через SFTP не масштабируется при увеличении числа внешних партнёров. При подключении дополнительных партнёров потребуется пересмотр архитектуры (переход на API) |
| Расширение адаптера сообщений | Добавление REST API и файлового экспорта в адаптер сообщений увеличивает его ответственность. В перспективе может потребоваться выделение в отдельный сервис |
| Нагрузка на команду АБС | Команда АБС (20 человек) параллельно разрабатывает функционал MVP (адаптер сообщений, управление ставками) и новый функционал (API, SFTP-экспорт). Необходимо планирование приоритетов |
